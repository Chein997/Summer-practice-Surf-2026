# BUG-002. При запросе SMS с телефоном `+abc` backend возвращает 500 вместо ошибки валидации

Проект: мобильное приложение картинг-центра «Апекс»  
Проверяемый компонент: backend API  
Среда проверки: локальный запуск backend на `http://127.0.0.1:8082`  
База данных: PostgreSQL в Docker на `127.0.0.1:5434`  
Дата проверки: 2026-07-06

## 1. Симптом

При запросе SMS с некорректным телефоном:

```text
+abc
```

backend возвращает:

```http
HTTP/1.1 500 Internal Server Error
```

Тело ответа:

```json
{
  "code": "INTERNAL_ERROR",
  "message": "Не удалось отправить SMS-код."
}
```

Проблема в том, что пользователь ввёл некорректный телефон, но API сообщает о внутренней ошибке сервера.

## 2. Требование

Тест-кейс: `TC-AUTH-003 — Валидация некорректного формата телефона`.

Ожидание по тест-кейсу:

```text
1. Ввести значение 123 или текст вместо телефона.
2. Нажать получение SMS-кода.
```

Ожидаемый результат:

```text
SMS-запрос не отправляется, приложение показывает ошибку формата телефона.
```

Для backend API это должно означать:

```http
HTTP/1.1 400 Bad Request
```

или:

```http
HTTP/1.1 422 Unprocessable Entity
```

с телом:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Введите телефон в международном формате."
}
```

## 3. Как получить баг

### 3.1. Подготовить переменную

```bash
BASE_URL="http://127.0.0.1:8082"
```

### 3.2. Отправить запрос SMS с телефоном `+abc`

```bash
curl -i -X POST "$BASE_URL/auth/request-sms" \
  -H "Content-Type: application/json" \
  -d '{"phone":"+abc"}'
```

### 3.3. Фактический результат

```http
HTTP/1.1 500 Internal Server Error
```

```json
{
  "code": "INTERNAL_ERROR",
  "message": "Не удалось отправить SMS-код."
}
```

## 4. Из-за чего происходит

В текущей реализации проверка телефона в `handleRequestSMS` слишком слабая:

```go
if req.Phone == "" || !strings.HasPrefix(req.Phone, "+") {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Введите телефон в международном формате.")
	return
}
```

Значение:

```text
+abc
```

проходит эту проверку, потому что начинается с `+`.

После этого backend пытается вставить запись в таблицу `otp_codes`:

```sql
INSERT INTO otp_codes (phone, purpose, code_hash, expires_at)
VALUES ($1, 'LOGIN', $2, $3)
```

На уровне PostgreSQL есть constraint на формат телефона. БД отклоняет значение `+abc`, а backend маппит эту ошибку в общий `INTERNAL_ERROR`.

Итог: ошибка пользовательского ввода превращается в `500`.

## 5. Фикс

### 5.1. Использовать общий валидатор телефона

Создать файл:

```text
backend/internal/server/validation.go
```

Содержимое:

```go
package server

import (
	"regexp"
	"strings"
)

var phoneRegexp = regexp.MustCompile(`^\+[1-9]\d{7,14}$`)

func isValidPhone(value string) bool {
	value = strings.TrimSpace(value)
	return phoneRegexp.MatchString(value)
}
```

Если файл `validation.go` уже создан при исправлении `BUG-001`, отдельно создавать его не нужно.

### 5.2. Исправить `handleRequestSMS`

В файле:

```text
backend/internal/server/auth.go
```

заменить:

```go
if req.Phone == "" || !strings.HasPrefix(req.Phone, "+") {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Введите телефон в международном формате.")
	return
}
```

на:

```go
if !isValidPhone(req.Phone) {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Введите телефон в международном формате.")
	return
}
```

### 5.3. Исправить `handleVerifySMS`

В `handleVerifySMS` сейчас проверяется только то, что телефон и код не пустые:

```go
if req.Phone == "" || req.Code == "" {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Введите телефон и SMS-код.")
	return
}
```

Лучше разделить проверки:

```go
if !isValidPhone(req.Phone) {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Введите телефон в международном формате.")
	return
}

if req.Code == "" {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Введите SMS-код.")
	return
}
```

Так endpoint проверки SMS тоже не будет работать с некорректным телефоном.

### 5.4. Дополнительная защита

Даже после application-level validation полезно отдельно маппить ошибки PostgreSQL constraint violations в `VALIDATION_ERROR`.

Например:

```go
var pgErr *pgconn.PgError
if errors.As(err, &pgErr) && pgErr.Code == "23514" {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Некорректный формат телефона.")
	return
}
```

Это нужно как safety-net, чтобы пользовательский ввод не превращался в `500`.
