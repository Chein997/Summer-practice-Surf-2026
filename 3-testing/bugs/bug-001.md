# BUG-001. При бронировании с email `ivan@` backend возвращает 500 вместо ошибки валидации

Проект: мобильное приложение картинг-центра «Апекс»  
Проверяемый компонент: backend API  
Среда проверки: локальный запуск backend на `http://127.0.0.1:8082`  
База данных: PostgreSQL в Docker на `127.0.0.1:5434`  
Дата проверки: 2026-07-06

## 1. Симптом

При создании брони с некорректным email backend возвращает:

```http
HTTP/1.1 500 Internal Server Error
```

Тело ответа:

```json
{
  "code": "INTERNAL_ERROR",
  "message": "Не удалось создать бронь."
}
```

Проблема в том, что пользователь получает внутреннюю ошибку сервера, хотя это обычная ошибка ввода данных.

## 2. Требование

Тест-кейс: `TC-BOOK-006 — Валидация некорректного email`.

Ожидание по тест-кейсу:

```text
1. Ввести ivan@ или ivan в email.
2. Заполнить остальные поля валидно.
3. Отправить форму.
```

Ожидаемый результат:

```text
Форма не отправляется; приложение показывает ошибку формата email.
```

Для backend API это должно означать:

```http
HTTP/1.1 400 Bad Request
```

или:

```http
HTTP/1.1 422 Unprocessable Entity
```

с телом:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "Введите корректный email."
}
```

## 4. Из-за чего происходит

В текущей реализации валидация email слишком слабая.

В функции `validateCreateBooking` проверяется только наличие символа `@`:

```go
if !strings.Contains(strings.TrimSpace(req.Profile.Email), "@") {
    return "VALIDATION_ERROR", "Введите корректный email."
}
```

Из-за этого значение:

```text
ivan@
```

проходит application-level validation, потому что символ `@` есть.

Дальше backend пытается вставить бронь в таблицу `bookings`. В БД есть более строгое ограничение на email, поэтому PostgreSQL отклоняет insert. Ошибка БД попадает в общий обработчик:

```go
s.logger.Error("insert booking failed", "error", internalErr(err))
writeError(w, http.StatusInternalServerError, "INTERNAL_ERROR", "Не удалось создать бронь.")
```

Итог: пользователь получает `500`, хотя должен получить понятную ошибку валидации.

## 5. Фикс

### 5.1. Добавить общий файл валидации

Создать файл:

```text
backend/internal/server/validation.go
```

Содержимое:

```go
package server

import (
	"regexp"
	"strings"
)

var (
	phoneRegexp = regexp.MustCompile(`^\+[1-9]\d{7,14}$`)
	emailRegexp = regexp.MustCompile(`^[A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Za-z]{2,}$`)
)

func isValidPhone(value string) bool {
	value = strings.TrimSpace(value)
	return phoneRegexp.MatchString(value)
}

func isValidEmail(value string) bool {
	value = strings.TrimSpace(value)
	return emailRegexp.MatchString(value)
}
```

### 5.2. Исправить проверку email в `bookings.go`

В файле:

```text
backend/internal/server/bookings.go
```

заменить:

```go
if !strings.Contains(strings.TrimSpace(req.Profile.Email), "@") {
    return "VALIDATION_ERROR", "Введите корректный email."
}
```

на:

```go
if !isValidEmail(req.Profile.Email) {
	return "VALIDATION_ERROR", "Введите корректный email."
}
```

Также желательно заменить проверку телефона в профиле бронирования:

```go
if !strings.HasPrefix(strings.TrimSpace(req.Profile.Phone), "+") {
	return "VALIDATION_ERROR", "Введите телефон в международном формате."
}
```

на:

```go
if !isValidPhone(req.Profile.Phone) {
	return "VALIDATION_ERROR", "Введите телефон в международном формате."
}
```

### 5.3. Дополнительная защита

Даже после application-level validation полезно отдельно маппить ошибки constraint violations PostgreSQL в `VALIDATION_ERROR`.

Например, если используется `pgconn.PgError`, можно обработать код `23514`:

```go
var pgErr *pgconn.PgError
if errors.As(err, &pgErr) && pgErr.Code == "23514" {
	writeError(w, http.StatusUnprocessableEntity, "VALIDATION_ERROR", "Некорректные данные бронирования.")
	return
}
```

Это не заменяет валидацию, но защищает API от возврата `500` при ошибках пользовательского ввода.
