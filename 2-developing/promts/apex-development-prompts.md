# Этап 1. Разбор репозитория и рамки разработки

## Цель

Понять текущую структуру репозитория и определить, что нужно реализовать для MVP клиентского приложения и API.

## Промт

```text
https://github.com/Chein997/Summer-practice-Surf-2026
Изучи репозиторий и текущие материалы проекта. Определи, что уже есть по аналитике, дизайну, backend и client, и составь план разработки MVP приложения картинг-центра «Апекс».
```

## Результат

- Зафиксирована структура проекта: `1-analitics`, `backend`, `client`.
- Определён фокус разработки: клиентское приложение + клиентский API для MVP.
- Подтверждён стек: backend на Go, client на Kotlin Compose Multiplatform.
- MVP ограничен клиентскими сценариями: SMS-авторизация, просмотр слотов, бронирование, мои брони, отмена брони, push-уведомления.

# Этап 8. Фикс багов бизнес-логики API

## Цель

Понять, какие изменения внести в backend, чтобы исправить найденные дефекты.

## Промт

```text
Нужно пофиксить эти баги, расскажи, что нужно делать.
```

## Результат

- Сформирован план исправлений backend.
- Для ошибки валидации телефона предложено:
  - валидировать формат телефона до отправки SMS;
  - возвращать клиентскую ошибку `400 Bad Request`, а не `500 Internal Server Error`;
  - не маскировать ошибку валидации как ошибку отправки SMS.
- Для ошибок бронирования предложено:
  - вынести проверки валидации в единый слой перед созданием брони;
  - проверять возраст участника;
  - проверять согласие с правилами безопасности;
  - проверять доступность слота и количество свободных мест;
  - проверять допустимость отмены относительно порога в 1 час;
  - возвращать понятные коды ошибок в едином формате API.

# Этап 9. Ручная проверка после фиксов

## Цель

Проверить, что исправления не сломали основной пользовательский сценарий.

## Промт

```text
Составь короткий чек-лист ручной проверки после фикса багов: авторизация, слоты, бронирование, валидация формы, отмена брони и ошибки API.
```

## Результат

- Подготовлен smoke-чек-лист:
  - валидный телефон принимает SMS-запрос;
  - невалидный телефон возвращает `400`, а не `500`;
  - SMS-код подтверждает авторизацию;
  - список слотов загружается;
  - недоступный слот нельзя забронировать;
  - пользователь младше 16 лет не может создать бронь;
  - бронь без согласия с правилами не создаётся;
  - валидная бронь создаётся и появляется в «Моих бронях»;
  - бронь можно отменить только при соблюдении правила `больше 1 часа до начала`;
  - ошибки возвращаются в едином формате.
