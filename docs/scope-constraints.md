# Ограничения ТЗ и границы скоупа MVP

Документ фиксирует ограничения реализации MVP мобильного приложения картинг-центра «Апекс» и перечень функций, которые не входят в текущий скоуп.

## 1. Общая граница MVP

MVP — это клиентское мобильное приложение для записи клиента на заезд в картинг-центре.

В MVP входят:

- авторизация клиента по номеру телефона и SMS-коду;
- просмотр доступных заездов на ближайшие 7 дней;
- просмотр деталей выбранного слота;
- создание бронирования на одно место;
- просмотр списка своих броней;
- просмотр деталей брони;
- отмена брони клиентом;
- отображение отмены заезда центром;
- регистрация push-токена устройства;
- получение push-уведомлений по ключевым событиям бронирования и заезда.

MVP не является системой управления картинг-центром. Приложение не должно управлять расписанием, трассами, маршалами, парком картов и административными действиями.

## 2. Ограничения по расписанию и слотам

### 2.1. Что нужно реализовать

Приложение должно показывать слоты заездов на ближайшие 7 дней.

Для каждого слота должны быть доступны основные данные:

- дата и время старта;
- длительность заезда;
- конфигурация трассы;
- уровень заезда;
- количество свободных мест;
- цена;
- статус;
- адрес картинг-центра;
- место встречи;
- правила безопасности;
- условия отмены;
- маршал, если назначен.

### 2.2. Статусы слота

| Статус | Описание | Можно бронировать |
|---|---|---|
| `AVAILABLE` | Есть свободные места. | Да |
| `NO_FREE_PLACES` | Свободных мест нет. | Нет |
| `CANCELLED` | Заезд отменён центром. | Нет |

### 2.3. Ограничения реализации

Backend является источником истины по:

- актуальности расписания;
- доступности слота;
- количеству свободных мест;
- статусу слота;
- возможности бронирования.

Клиентское приложение не должно самостоятельно принимать финальное решение о доступности бронирования. Оно может показывать состояние, полученное от backend, но backend обязан повторно проверять доступность при создании брони.

## 3. Ограничения по бронированию

### 3.1. Основные правила

- Одна бронь относится к одному слоту.
- В MVP одна бронь равна одному месту.
- Групповое бронирование не поддерживается.
- После создания бронь получает статус `PENDING_CONFIRMATION`.
- Подтверждение или отклонение брони выполняется администратором во внешней системе.
- Мобильное приложение не содержит админского подтверждения брони.
- Один клиент не может создать несколько активных или ожидающих подтверждения броней на один и тот же слот.

### 3.2. Обязательные данные для бронирования

Для создания брони обязательны:

- имя клиента;
- телефон;
- email;
- возраст;
- согласие с правилами безопасности.

Дополнительное правило:

- минимальный возраст клиента — 16 лет;
- если клиенту меньше 18 лет, требуется согласие родителя или законного представителя.

### 3.3. Что должен проверять backend

Backend при создании брони обязан проверить:

- слот существует;
- слот не отменён;
- в слоте есть свободные места;
- клиент не создаёт дубль брони на этот же слот;
- заполнены обязательные поля;
- возраст клиента не меньше 16 лет;
- согласие с правилами безопасности принято;
- для клиента младше 18 лет принято согласие родителя или законного представителя.

Клиентская валидация нужна только для UX и не заменяет серверные проверки.

## 4. Ограничения по статусам брони

В MVP поддерживаются статусы:

| Статус | Описание |
|---|---|
| `PENDING_CONFIRMATION` | Бронь создана и ожидает подтверждения администратором. |
| `ACTIVE` | Бронь подтверждена и активна. |
| `CANCELLED_BY_CLIENT` | Бронь отменена клиентом. |
| `CANCELLED_BY_CENTER` | Бронь отменена картинг-центром. |
| `REJECTED_BY_CENTER` | Бронь отклонена картинг-центром. |
| `COMPLETED` | Заезд завершён. |
| `NO_SHOW` | Клиент не пришёл на заезд. |

Старые или неподходящие статусы вроде `active`, `cancelled`, `late_cancel` не должны использоваться в новой доменной модели.

## 5. Ограничения по отмене брони клиентом

Клиент может отменить бронь только при выполнении всех условий:

- бронь принадлежит текущему клиенту;
- бронь находится в статусе `PENDING_CONFIRMATION` или `ACTIVE`;
- до начала заезда осталось больше 1 часа.

Если до старта заезда осталось 1 час или меньше, отмена клиентом недоступна.

При успешной отмене:

- статус брони меняется на `CANCELLED_BY_CLIENT`;
- фиксируется дата и время отмены;
- фиксируется источник отмены `CLIENT`;
- место в слоте освобождается.

Это правило должно реализовываться в backend service-слое в транзакции, а не только ограничениями PostgreSQL.

## 6. Ограничения по отмене заезда центром

Если заезд отменён центром:

- слот получает статус `CANCELLED`;
- бронирование получает статус `CANCELLED_BY_CENTER`;
- повторная запись на этот слот запрещается;
- клиенту показывается причина отмены;
- клиенту показываются дата и время отменённого заезда;
- клиенту доступно действие выбора другого заезда.

### 6.1. Причины отмены центром

| Код | Описание |
|---|---|
| `WEATHER` | Погодные условия. |
| `TECHNICAL_FAILURE` | Техническая неисправность. |
| `TRACK_UNAVAILABLE` | Недоступность трассы. |
| `MARSHAL_UNAVAILABLE` | Недоступность маршала. |
| `ORGANIZATIONAL` | Организационная причина. |
| `OTHER` | Другая причина со свободным текстом. |

Для причины `OTHER` должен быть указан текст причины.

## 7. Ограничения по авторизации

В MVP авторизация выполняется только по номеру телефона и SMS-коду.

Нужны сущности:

- `Client`;
- `OtpCode`;
- `AuthSession`.

Не нужно реализовывать:

- парольную авторизацию;
- авторизацию по email;
- авторизацию через социальные сети;
- восстановление пароля;
- полноценный личный кабинет с редактированием профиля.

## 8. Ограничения по push-уведомлениям

В MVP нужны push-уведомления по событиям:

- бронь подтверждена;
- бронь отклонена;
- напоминание за 24 часа до старта;
- напоминание за 2 часа до старта;
- заезд отменён центром.

В MVP не нужны push-уведомления:

- о создании брони в ожидании подтверждения;
- об отмене брони клиентом;
- о невозможности отмены из-за порога в 1 час.

Для реализации достаточно хранить push-токены устройств. Журнал push-событий можно добавить, если нужен аудит, отладка или очередь отправки.

## 9. Ограничения ответственности клиента и backend

### 9.1. CMP-клиент отвечает за

- отображение экранов;
- валидацию формы для удобства пользователя;
- отправку запросов в API;
- отображение loading/empty/error/success-состояний;
- отображение статусов слотов и броней;
- отображение сообщений об ошибках API;
- запрос разрешения на push-уведомления;
- регистрацию push-токена устройства.

### 9.2. Backend отвечает за

- проверку бизнес-правил;
- транзакционное создание брони;
- транзакционную отмену брони;
- защиту от двойного бронирования;
- актуальность свободных мест;
- статусы слотов;
- статусы броней;
- проверку возраста и согласий;
- проверку права клиента на просмотр и отмену брони;
- выдачу данных по OpenAPI-контракту.

### 9.3. Что нельзя перекладывать на клиент

Клиент не должен быть источником истины по:

- свободным местам;
- возможности бронирования;
- возможности отмены;
- статусу слота;
- статусу брони;
- подтверждению или отклонению брони;
- отмене заезда центром.

## 10. Что не входит в продуктовый скоуп MVP

В MVP не входят:

- онлайн-оплата;
- возвраты платежей;
- заказы и корзина;
- групповое бронирование;
- выбор конкретного карта;
- выбор экипировки;
- фильтры по дате;
- фильтры по времени;
- фильтры по трассе;
- оценка маршала;
- рейтинг маршалов;
- программа лояльности;
- промокоды;
- бонусы;
- абонементы;
- автоматическое подтверждение брони;
- тайм-аут подтверждения брони;
- история поездок вне списка броней;
- чат с администратором.

## 11. Что не входит в клиентское приложение

Мобильное приложение не должно содержать:

- админскую панель;
- создание слотов;
- редактирование слотов;
- удаление слотов;
- управление расписанием;
- назначение маршалов;
- управление маршалами;
- управление трассами;
- управление конфигурациями трасс;
- управление парком картов;
- подтверждение брони администратором;
- отклонение брони администратором;
- отмену слота администратором;
- управление причинами отмены;
- миграцию исторических данных.

Эти функции относятся к существующей внешней системе или к будущим релизам.

## 12. Что не входит в UI MVP

В UI MVP не нужно проектировать:

- экран оплаты;
- экран выбора экипировки;
- экран выбора карта;
- экран рейтинга маршала;
- экран программы лояльности;
- экран промокодов;
- экран администрирования;
- экран управления расписанием;
- фильтры расписания;
- время, к которому нужно приехать заранее.

## 13. Ограничения для PostgreSQL-модели

### 13.1. Таблицы в скоупе

В скоупе PostgreSQL-модели:

- `clients`;
- `otp_codes`;
- `auth_sessions`;
- `track_configurations`;
- `ride_levels`;
- `marshals`;
- `ride_slots`;
- `bookings`;
- `push_device_tokens`.

Опционально:

- `push_notification_events`;
- `idempotency_keys`.

### 13.2. Таблицы вне скоупа

Не нужно добавлять таблицы:

- `payments`;
- `orders`;
- `group_bookings`;
- `equipment`;
- `kart_selection`;
- `karts`;
- `kart_fleet`;
- `loyalty_accounts`;
- `loyalty_transactions`;
- `marshal_reviews`;
- `admin_users`;
- `admin_actions`;
- `schedule_editor`;
- `fleet_management`.

### 13.3. Важные ограничения БД

В БД желательно зафиксировать:

- формат телефона;
- обязательность профиля бронирования;
- минимальный возраст 16 лет;
- обязательность согласия с правилами безопасности;
- обязательность родительского согласия для клиентов младше 18 лет;
- допустимые статусы слотов;
- допустимые статусы броней;
- допустимые причины отмены центром;
- уникальность активной или ожидающей подтверждения брони клиента на один слот;
- корректность количества свободных мест;
- запрет пустых текстовых обязательных полей.

## 14. Ограничения для backend API

Backend API должен покрывать только следующие группы сценариев:

### Auth

- запрос SMS-кода;
- подтверждение SMS-кода.

### RideSlots

- получение списка слотов;
- получение деталей слота.

### Bookings

- создание брони;
- получение списка моих броней;
- получение деталей брони;
- отмена брони клиентом.

### Push

- регистрация push-токена;
- удаление push-токена.

Не нужно добавлять endpoint-ы для оплаты, администрирования, управления расписанием, выбора карта, выбора экипировки, рейтингов и лояльности.

## 15. Ограничения для CMP-клиента

CMP-клиент должен быть построен вокруг клиентских сценариев:

- ввод номера телефона;
- ввод SMS-кода;
- список доступных слотов;
- пустое расписание;
- детали слота;
- форма бронирования;
- бронь создана и ожидает подтверждения;
- мои брони;
- детали брони;
- подтверждение отмены брони клиентом;
- заезд отменён центром;
- ошибка API или недоступность сервиса;
- разрешение push-уведомлений.

Не нужно добавлять клиентские feature-модули для:

- оплаты;
- экипировки;
- выбора карта;
- рейтингов;
- лояльности;
- администрирования;
- управления расписанием.

## 16. Чеклист проверки скоупа

Перед добавлением новой сущности, таблицы, endpoint-а, экрана или use case нужно проверить:

```text
1. Это нужно для авторизации, просмотра слотов, бронирования, отмены или push?
2. Это есть в ТЗ, требованиях, app-spec или OpenAPI?
3. Это клиентский сценарий, а не админское действие?
4. Это не относится к оплате, экипировке, выбору карта, рейтингам или лояльности?
5. Backend остаётся источником истины?
6. Бронь всё ещё создаётся только на одно место?
7. После создания бронь получает статус PENDING_CONFIRMATION?
8. Клиент не подтверждает и не отклоняет бронь самостоятельно?
9. Отмена клиентом запрещена за 1 час или меньше до старта?
10. Слот с NO_FREE_PLACES или CANCELLED недоступен для бронирования?
```

Если новая функция не проходит этот чеклист, её не нужно включать в MVP.
