# Консолидация промтов по анализу проекта «Апекс»

Назначение файла: компактно собрать промты из рабочих чатов по анализу и проектированию MVP мобильного приложения картинг-центра «Апекс»: от брифа и домена до требований, экранов, модели данных, OpenAPI и ТЗ.


## Общие ограничения для всех этапов

- Проект: MVP клиентского мобильного приложения картинг-центра «Апекс».
- Не добавлять того, чего нет в домене и требованиях.
- Не смешивать бизнес-требования, функциональные требования и нефункциональные требования.
- Результаты оформлять в отдельные markdown/yaml-файлы.
- Существующий backend считать источником истины по расписанию, слотам, доступности, статусам и бизнес-ограничениям.
- Клиентское приложение не управляет расписанием, трассами, маршалами, парком картов и административными действиями.
- Вне MVP: онлайн-оплата, групповое бронирование, выбор карта/экипировки, фильтры, рейтинг маршала, программа лояльности, промокоды, админка, автоматическое подтверждение брони.

# Этап 1. Бриф, вопросы заказчику и описание домена

## Цель

Из брифа получить список вопросов заказчику, закрыть открытые моменты и сформировать доменное описание MVP.

## Промты

```text
Изучи бриф и сформируй вопросы
```

```text
создай два файла customer-questions.md и domain-description.md и вставь туда соответствующие пункты
```

```text
давай в режиме вопрос-ответ я отвечу на открытые вопросы
```

```text
Первый пункт. Мы вышли за рамки вопросов которые описаны в customer-questions.md. Обнови список вопросов в файле.
```

```text
Проверь, может какие-то уточнения есть и для domain-description
```

## Ключевые решения по результатам ответов

- Для бронирования обязательны: имя, телефон, email, возраст, согласие с правилами безопасности.
- Минимальный возраст участника: 16 лет.
- Согласие с правилами безопасности обязательно.
- Время, к которому нужно приехать заранее, исключено как лишнее для MVP.
- Порог клиентской отмены брони: больше 1 часа до начала заезда.
- В MVP остаются клиентские сценарии: просмотр слотов, детали слота, бронирование, мои брони, отмена брони, статусы/ошибки, push-уведомления.

## Результат

- `customer-questions.md` — список вопросов заказчику.
- `domain-description.md` — описание домена и границ MVP.

```text
Из domain-description.md, business-requirements.md, functional-requirements.md, non-functional-requirements.md, use-cases.md и user-stories.md собери концептуальную ER/API-модель клиентского мобильного приложения «Апекс».

Нужно:
```

# Этап 5. OpenAPI-спецификация

## Цель

На основе требований, модели данных и постановок экранов сформировать клиентский API-контракт MVP.

## Промты

```text
сгенерируй openAPI спецификацию. результат в отдельном файлике
```

## Реконструированный уточняющий промт

```text
Используй требования, экранные постановки, data-model.md и api-sequence.md. Сформируй OpenAPI 3.1 спецификацию для MVP клиентского мобильного приложения «Апекс».

Включи API для:
- авторизации по телефону и SMS-коду;
- получения списка слотов;
- получения деталей слота;
- создания брони;
- просмотра моих броней;
- просмотра деталей брони;
- отмены брони клиентом;
- регистрации и удаления push device token.

Сохрани единый формат ошибок. Админские действия, онлайн-оплату, групповое бронирование, фильтры и управление расписанием не добавлять.
```

## Результат

- `openapi-apex-mobile.yaml`

Ключевые группы API:

- `Auth` — `POST /auth/request-sms`, `POST /auth/verify-sms`.
- `RideSlots` — список и детали слотов.
- `Bookings` — создание, просмотр и отмена брони.
- `Push` — регистрация/удаление device token.
