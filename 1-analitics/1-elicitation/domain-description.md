# Описание домена

Документ сформирован на основе брифа по клиентскому мобильному приложению картинг-центра «Апекс» и уточнений, полученных в Q&A-сессии с заказчиком.

## 1. Краткое описание домена

Домен — клиентская запись на заезды в уличном картинг-центре.

Картинг-центр проводит групповые заезды по заранее сформированному расписанию. Каждый заезд проходит в конкретное время, на конкретной конфигурации трассы, с ограниченным числом мест и под контролем маршала-инструктора.

Клиент через мобильное приложение просматривает доступные слоты, открывает детали заезда, авторизуется по номеру телефона, создаёт бронь, отслеживает её статус, получает push-уведомления и при необходимости отменяет бронь по правилам центра.

В текущем скоупе мобильное приложение не управляет расписанием, маршалами, трассами, парком картов и административными действиями. Эти данные и действия находятся в существующей инфраструктуре. Бэкенд является источником истины: он отвечает за доступность мест, отсутствие двойных броней, статусы слотов, ручное подтверждение броней администратором и отмену заездов центром. Клиентское приложение должно корректно отображать данные, отправлять действия клиента и обрабатывать ответы API.

## 2. Границы MVP

### 2.1. Входит в MVP

В первую поставку входят:

1. просмотр доступных слотов;
2. просмотр деталей слота;
3. регистрация / авторизация клиента;
4. бронирование места;
5. просмотр своих броней;
6. отмена брони клиентом;
7. push-уведомления;
8. отображение отмены заезда центром.

### 2.2. Отложено за пределы MVP

В текущую поставку не входят и могут быть реализованы позже:

1. фильтры по дате / времени / трассе;
2. выбор своей или прокатной экипировки;
3. оценка маршала после заезда;
4. расширенная программа лояльности;
5. онлайн-оплата.

## 3. Основные роли

| Роль | Описание | В текущем скоупе |
|---|---|---|
| Клиент | Пользователь мобильного приложения, который просматривает слоты, бронирует место, отслеживает статус брони и может отменить бронь | Да |
| Маршал-инструктор | Проводит инструктаж и контролирует заезд | Нет, интерфейс уже существует |
| Владелец / администратор | Управляет расписанием, слотами, отменами, персоналом, подтверждает или отклоняет брони | Нет, админка уже существует |
| Существующий бэкенд | Источник расписания, слотов, статусов, доступности и бизнес-ограничений | Да, через API |

## 4. Ключевые сущности домена

| Сущность | Что означает |
|---|---|
| Клиент | Пользователь мобильного приложения, который авторизуется по телефону и бронирует участие в заездах |
| Профиль клиента | Набор данных клиента: имя, телефон, email, возраст, согласие с правилами безопасности, а при необходимости согласие родителя / законного представителя |
| Заезд / слот | Конкретный старт в расписании: дата, время, трасса, уровень, маршал, вместимость, свободные места, цена, статус |
| Бронь | Запись клиента на конкретный заезд; создаётся в статусе «Ожидает подтверждения» и затем подтверждается или отклоняется администратором |
| Карт | Машина для участия в заезде; количество картов ограничивает вместимость заезда |
| Конфигурация трассы | Вариант трассы, например короткая для новичков или длинная для опытных |
| Уровень заезда | Классификация слота по сложности: новичковый или опытный |
| Маршал | Инструктор, отвечающий за безопасность и проведение заезда |
| Отмена клиентом | Действие клиента по отмене своей брони, доступное только если до старта осталось больше 1 часа |
| Отмена центром | Отмена заезда со стороны центра с указанием причины; клиент видит статус, причину, дату и время заезда, а также кнопку выбора другого заезда |
| Push-уведомление | Сообщение клиенту о подтверждении брони, отклонении брони, напоминании о заезде или отмене заезда центром |

## 5. Авторизация и данные клиента

Клиент входит в приложение по номеру телефона с подтверждением через SMS-код.

Для бронирования обязательны следующие данные:

1. имя;
2. телефон;
3. email;
4. возраст;
5. согласие с правилами безопасности.

Для участия действует минимальный возраст — 16 лет. Если клиенту меньше 18 лет, требуется согласие родителя или законного представителя.

## 6. Слоты и их отображение

По умолчанию клиенту показываются слоты на ближайшие 7 дней от текущего момента. Если расписания на ближайшие дни нет, приложение показывает понятное пустое состояние, например «Пока нет доступных заездов».

### 6.1. Список слотов

В списке слотов клиент видит:

1. дату и время старта;
2. длительность заезда;
3. конфигурацию трассы: короткая / длинная;
4. уровень: новичковый / опытный;
5. количество свободных мест;
6. цену;
7. статус слота: доступен / мест нет / отменён;
8. адрес картинг-центра.

Слоты без свободных мест показываются со статусом «Мест нет» и без возможности бронирования.

Слоты, отменённые центром, показываются со статусом «Отменён» и без возможности бронирования.

### 6.2. Детали слота

На экране деталей слота клиент видит:

1. дату и время старта;
2. длительность заезда;
3. конфигурацию трассы;
4. уровень сложности;
5. количество свободных мест;
6. цену;
7. маршала-инструктора;
8. адрес центра;
9. место сбора на территории;
10. правила безопасности;
11. условия отмены;
12. кнопку «Забронировать».

Время, к которому нужно приехать заранее, в MVP не показывается.

## 7. Бронирование

Один аккаунт клиента может забронировать только одно место в одном заезде. Групповое бронирование в MVP не поддерживается.

После нажатия кнопки «Забронировать» приложение отправляет запрос в API. Бэкенд проверяет доступность места и создаёт бронь либо возвращает отказ, если бронь невозможна.

Бронь не подтверждается автоматически. После создания она получает статус «Ожидает подтверждения». Администратор вручную подтверждает или отклоняет бронь в существующей админке.

Если администратор подтверждает бронь, клиент получает push-уведомление, а статус брони становится «Активна».

Если администратор отклоняет бронь, клиент получает push-уведомление, а статус брони становится «Отклонена центром».

Если администратор долго не подтверждает бронь, она остаётся в статусе «Ожидает подтверждения». Автоматического тайм-аута подтверждения в MVP нет.

## 8. Статусы брони

В MVP используются следующие статусы брони:

| Статус | Значение |
|---|---|
| Ожидает подтверждения | Бронь создана клиентом и ожидает ручного решения администратора |
| Активна | Бронь подтверждена, клиент записан на будущий заезд |
| Отменена клиентом | Клиент отменил бронь по правилам центра |
| Отменена центром | Заезд отменён центром после создания брони |
| Отклонена центром | Администратор отказал в подтверждении брони |
| Завершена | Заезд прошёл |
| Неявка | Клиент не приехал на заезд |

## 9. Раздел «Мои брони»

В приложении нужен отдельный раздел «Мои брони».

Раздел предназначен для просмотра клиентом собственных броней и их актуальных статусов.

Детализация раздела, группировка броней и состав карточки брони требуют отдельного уточнения, так как в Q&A-сессии был зафиксирован только сам факт наличия раздела.

## 10. Отмена брони клиентом

Клиент может отменить бронь через приложение, если до старта заезда осталось больше 1 часа.

Правила отмены:

1. более чем за 1 час до старта — отмена разрешена;
2. за 1 час или меньше до старта — отмена через приложение запрещена;
3. при успешной отмене место сразу освобождается;
4. условия отмены показываются клиенту перед бронированием.

## 11. Отмена заезда центром

Если заезд отменяет центр, бронь не удаляется, а получает статус «Отменена центром».

В приложении клиент видит:

1. статус брони «Отменена центром»;
2. причину отмены;
3. дату и время отменённого заезда;
4. кнопку «Выбрать другой заезд».

Отдельный текст «Повторная запись на этот слот недоступна» в MVP не показывается.

Поддерживаются следующие причины отмены центром:

1. погодные условия;
2. техническая неисправность;
3. недоступность трассы;
4. недоступность маршала;
5. организационная причина;
6. другое — свободный текст от администратора.

## 12. Push-уведомления

В MVP нужны push-уведомления по следующим событиям:

1. бронь подтверждена администратором;
2. бронь отклонена администратором;
3. напоминание о заезде;
4. заезд отменён центром.

Напоминания о заезде отправляются несколько раз:

1. за 24 часа до старта;
2. за 2 часа до старта.

В MVP не отправляются push-уведомления о том, что бронь создана и ожидает подтверждения, что бронь отменена клиентом, а также о том, что отмена запрещена из-за порога в 1 час.

## 13. Основной жизненный цикл брони

1. Клиент открывает приложение.
2. Клиент авторизуется по номеру телефона с SMS-кодом, если ещё не авторизован.
3. Приложение получает из API доступные слоты, по умолчанию на ближайшие 7 дней.
4. Клиент выбирает слот из списка.
5. Клиент открывает детали слота и видит условия заезда, цену, правила безопасности и условия отмены.
6. Клиент нажимает «Забронировать».
7. Бэкенд проверяет доступность места и создаёт бронь в статусе «Ожидает подтверждения» либо возвращает отказ.
8. Администратор подтверждает или отклоняет бронь в существующей админке.
9. При подтверждении бронь становится активной, клиент получает push-уведомление.
10. При отклонении бронь получает статус «Отклонена центром», клиент получает push-уведомление.
11. До старта клиент может отменить активную или ожидающую бронь, если до старта осталось больше 1 часа.
12. Если заезд отменяет центр, бронь получает статус «Отменена центром», клиент получает push-уведомление и может выбрать другой заезд.
13. После проведения заезда бронь получает статус «Завершена» или «Неявка» на стороне существующей инфраструктуры.

## 14. Важные бизнес-правила

| Правило | Описание |
|---|---|
| Расписание формируется заранее | Клиентское приложение только читает расписание из API |
| Горизонт по умолчанию — 7 дней | Клиенту по умолчанию показываются слоты на ближайшие 7 дней |
| Вместимость ограничена | Количество мест зависит от картов, типа заезда и ограничений безопасности |
| Двойные брони недопустимы | Гарантируется бэкендом, клиент обрабатывает отказ |
| Один аккаунт — одно место в заезде | Групповое бронирование в MVP не поддерживается |
| Минимальный возраст — 16 лет | Клиент младше 16 лет не может записаться на заезд |
| Для клиентов младше 18 лет требуется согласие | Нужно согласие родителя или законного представителя |
| Бронь требует ручного подтверждения | После создания бронь получает статус «Ожидает подтверждения» |
| Автоматического тайм-аута подтверждения нет | Если администратор долго не подтверждает бронь, она остаётся в ожидании |
| Отмена клиентом ограничена по времени | Клиент может отменить бронь только более чем за 1 час до старта |
| При клиентской отмене место освобождается | После успешной отмены место снова становится доступным |
| Отмена центром не удаляет бронь | Бронь получает статус «Отменена центром» и причину |
| Повторная запись на отменённый слот запрещена | Отменённый слот показывается как недоступный для бронирования |
| Погода влияет на проведение | При опасных условиях центр может отменить заезд целиком |
| Экипировка отложена | Выбор собственной или прокатной экипировки не входит в MVP |
| Оценка маршала отложена | Клиентская оценка маршала не входит в MVP |

## 15. Граница серверной интеграции

Существующий бэкенд рассматривается как black-box источник истины.

Вне скоупа клиентского приложения находятся:

1. серверная транзакционность;
2. SLA бэкенда;
3. маппинг API на внутренние модели;
4. формирование расписания;
5. назначение маршалов;
6. управление парком картов;
7. отмена слота со стороны администратора;
8. миграция и дозаполнение исторических данных.

Со стороны клиента эта граница закрывается контрактом API. Приложение полагается на ответы бэкенда и корректно обрабатывает отказ при отсутствии мест, отменённом слоте, отклонении брони или недоступности действия.

## 16. Итоговая формулировка

Клиентское приложение картинг-центра — это мобильный сервис записи на групповые заезды, который позволяет клиенту просматривать слоты, открывать детали заезда, авторизоваться по телефону, создавать бронь, видеть её статус, получать важные push-уведомления и отменять бронь по правилам центра.

Система работает поверх существующего бэкенда, который хранит расписание, проверяет доступность мест, обеспечивает отсутствие двойных броней, передаёт статусы слотов и броней, а также принимает результаты административных действий из существующей инфраструктуры.
