# Вопросы к заказчику

Документ сформирован на основе брифа по клиентскому мобильному приложению картинг-центра «Апекс» и текущей Q&A-сессии с заказчиком.

## 1. Закрытые вопросы и зафиксированные ответы

### 1.1. Границы MVP и приоритеты

**Вопрос:** что точно должно войти в первую поставку за два месяца?

**Ответ:** в MVP обязательно входят:

1. просмотр доступных слотов;
2. просмотр деталей слота;
3. регистрация / авторизация клиента;
4. бронирование места;
5. просмотр своих броней;
6. отмена брони клиентом;
7. push-уведомления;
8. отображение отмены заезда центром.

**Можно отложить:**

- фильтры по дате / времени / трассе;
- выбор своей или прокатной экипировки;
- оценку маршала после заезда.

### 1.2. Регистрация и авторизация

**Вопрос:** как клиент должен входить в приложение?

**Ответ:** авторизация выполняется по номеру телефона с SMS-кодом.

**Вопрос:** какие данные клиента обязательны для бронирования?

**Ответ:** обязательны:

- имя;
- телефон;
- email;
- возраст;
- согласие с правилами безопасности.

### 1.3. Возрастные ограничения

**Вопрос:** есть ли ограничения для записи?

**Ответ:** да, действует минимальный возраст для любого заезда.

**Вопрос:** какой минимальный возраст?

**Ответ:** минимальный возраст — 16 лет.

**Вопрос:** если клиент младше 18 лет, нужно ли согласие родителя / законного представителя?

**Ответ:** да, согласие необходимо.

### 1.4. Бронирование нескольких мест

**Вопрос:** может ли один клиент забронировать несколько мест в одном заезде для компании?

**Ответ:** нет. Один аккаунт клиента может забронировать только одно место в одном заезде. Групповое бронирование в MVP не поддерживается.

### 1.5. Список доступных слотов

**Вопрос:** какие данные клиент должен видеть сразу в списке заездов?

**Ответ:** в списке доступных слотов обязательно показывать:

1. дату и время старта;
2. длительность заезда;
3. конфигурацию трассы: короткая / длинная;
4. уровень: новичковый / опытный;
5. количество свободных мест;
6. цену;
7. статус слота: доступен / мест нет / отменён;
8. адрес картинг-центра.

### 1.6. Экран деталей слота

**Вопрос:** что должно быть на экране деталей заезда перед бронированием?

**Ответ:** на экране деталей слота показываем:

1. дату и время старта;
2. длительность заезда;
3. конфигурацию трассы;
4. уровень сложности;
5. количество свободных мест;
6. цену;
7. маршала-инструктора;
8. адрес центра;
9. место сбора на территории;
10. правила безопасности;
11. условия отмены;
12. кнопку «Забронировать».

**Исключено:** время, к которому нужно приехать заранее, не показываем.

### 1.7. Статусы брони

**Вопрос:** какие статусы брони нужны в MVP?

**Ответ:** в MVP используются следующие статусы брони:

1. **Ожидает подтверждения** — бронь создана, но ещё не подтверждена администратором;
2. **Активна** — бронь подтверждена, клиент записан на будущий заезд;
3. **Отменена клиентом**;
4. **Отменена центром** — заезд отменён центром;
5. **Отклонена центром** — администратор отказал в подтверждении брони;
6. **Завершена** — заезд прошёл;
7. **Неявка** — клиент не приехал.

### 1.8. Отмена брони клиентом

**Вопрос:** за сколько времени до старта клиент может отменить бронь через приложение?

**Ответ:** клиент может отменить бронь через приложение, если до старта осталось больше 1 часа.

**Правило:**

- более чем за 1 час до старта — отмена разрешена;
- за 1 час или меньше до старта — отмена через приложение запрещена;
- при успешной отмене место сразу освобождается;
- условия отмены показываются клиенту перед бронированием.

### 1.9. Подтверждение брони

**Вопрос:** бронь создаётся сразу автоматически или требует подтверждения со стороны центра?

**Ответ:** бронь требует ручного подтверждения администратором.

**Правило:**

- после нажатия «Забронировать» создаётся бронь в статусе **«Ожидает подтверждения»**;
- администратор подтверждает или отклоняет бронь вручную в существующей админке;
- клиент должен видеть статус брони в приложении.

### 1.10. Действия после ручного подтверждения или отказа

**Вопрос:** что происходит после ручного подтверждения или отказа администратора?

**Ответ:**

- если администратор подтвердил бронь — клиент получает push-уведомление, статус становится **«Активна»**;
- если администратор отклонил бронь — клиент получает push-уведомление, статус становится **«Отклонена центром»**;
- если администратор долго не подтверждает бронь — она остаётся в ожидании, автоматического тайм-аута нет.

### 1.11. Push-уведомления

**Вопрос:** какие push-уведомления нужны в MVP?

**Ответ:** в MVP нужны push-уведомления:

1. бронь подтверждена администратором;
2. бронь отклонена администратором;
3. напоминание о заезде;
4. заезд отменён центром.

**Не включаем в MVP push:**

- бронь создана и ожидает подтверждения;
- бронь отменена клиентом;
- клиенту запрещена отмена из-за порога в 1 час.

### 1.12. Напоминания о заезде

**Вопрос:** когда отправлять push-напоминание о заезде?

**Ответ:** отправлять несколько напоминаний:

- первое — за 24 часа до старта;
- второе — за 2 часа до старта.

### 1.13. Отмена заезда центром

**Вопрос:** когда центр отменяет заезд, что клиент должен видеть в приложении?

**Ответ:** клиент видит:

1. статус брони **«Отменена центром»**;
2. причину отмены;
3. дату и время отменённого заезда;
4. кнопку **«Выбрать другой заезд»**.

**Не показываем:** отдельный текст «Повторная запись на этот слот недоступна».

### 1.14. Причины отмены центром

**Вопрос:** какие причины отмены нужно поддержать?

**Ответ:** поддерживаем все причины:

1. погодные условия;
2. техническая неисправность;
3. недоступность трассы;
4. недоступность маршала;
5. организационная причина;
6. другое — свободный текст от администратора.

### 1.15. Отображение недоступных слотов

**Вопрос:** что делать со слотами, на которые клиент уже не может записаться?

**Ответ:**

- слоты без свободных мест показываем со статусом **«Мест нет»**, без кнопки бронирования;
- слоты, отменённые центром, показываем со статусом **«Отменён»**, без кнопки бронирования.

### 1.16. Раздел «Мои брони»

**Вопрос:** как клиент должен видеть свои брони?

**Ответ:** нужен отдельный раздел **«Мои брони»**.

## 2. Открытые вопросы из исходного списка

Ниже оставлены вопросы, которые ещё не закрыты в Q&A-сессии и относятся к текущему скоупу клиентского приложения и API.

### 2.1. Границы продукта и платформа

1. Нужна ли веб-версия для клиентов или только мобильное приложение?
2. Приложение должно быть нативным iOS/Android или достаточно кроссплатформенного решения?
3. Есть ли фирменный стиль, логотип, цвета, тексты, tone of voice?

### 2.2. Пользовательский профиль

1. Нужен ли профиль клиента с историей бронирований?
2. Нужно ли хранить в профиле согласие с правилами безопасности и согласие родителя / законного представителя для клиентов младше 18 лет?

### 2.3. Просмотр слотов

1. Нужно ли показывать имя маршала в списке слотов или только на экране деталей?
2. Нужно ли показывать рейтинг маршала клиентам или рейтинг нужен только владельцу?
3. Что делать, если на ближайшие 7 дней нет доступных заездов: показывать только empty state или дать возможность оставить заявку?
4. Нужен ли лист ожидания, если мест нет?

### 2.4. Бронирование

1. Нужно ли ограничивать количество активных броней на одного клиента?
2. За сколько минут до старта можно создать новую бронь?
3. Что делать, если клиент выбрал слот, но пока подтверждал бронь, места закончились?
4. Нужна ли защита от повторного нажатия / дублей брони на клиенте?
5. Нужно ли отправлять клиенту номер брони или QR-код для регистрации на месте?

### 2.5. Экипировка

Экипировка отложена за пределы MVP, но вопросы стоит сохранить для будущего этапа:

1. Какие варианты экипировки доступны: свой шлем, свой подшлемник, прокат шлема, прокат подшлемника, полный комплект?
2. Экипировка влияет на цену брони?
3. Нужно ли клиенту указывать размеры экипировки?
4. Есть ли ограниченное количество прокатной экипировки по типам и размерам?
5. Что делать, если место в заезде есть, но нужной экипировки уже нет?
6. Можно ли изменить выбор экипировки после бронирования?

### 2.6. Отмена клиентом

1. Нужно ли клиенту указывать причину отмены?
2. Что делать с клиентами, которые часто отменяют поздно или не приезжают?

### 2.7. Отмена центром и погода

1. Текст причины отмены приходит из бэкенда или выбирается из фиксированного списка?
2. Должно ли приложение показывать погодное предупреждение до официальной отмены?
3. Нужно ли отображать статус «под вопросом из-за погоды»?
4. Какой канал уведомления обязателен при отмене, кроме push, если push отключён: SMS, email или ничего дополнительного?

### 2.8. Уведомления

1. Что делать, если push не доставлен или пользователь отключил push-уведомления?
2. Нужны ли уведомления внутри приложения — inbox / центр уведомлений?

### 2.9. Оценка маршала

Оценка маршала отложена за пределы MVP, но вопросы стоит сохранить для будущего этапа:

1. Оценка ставится по шкале 1–5, лайк/дизлайк или с текстовым отзывом?
2. Когда клиент может поставить оценку: сразу после времени окончания заезда, после отметки «заезд завершён» из бэкенда, в течение N дней?
3. Можно ли изменить оценку после отправки?
4. Нужно ли клиенту видеть, кому именно он ставит оценку?
5. Нужно ли собирать дополнительные параметры: безопасность, объяснение правил, доброжелательность, общее впечатление?
6. Нужна ли модерация текстовых отзывов, если они будут?

### 2.10. Оплата

1. Входит ли оплата в MVP или пока только бронирование без оплаты?
2. Если онлайн-оплаты пока нет, нужно ли показывать цену и способ оплаты на месте?
3. Нужно ли фиксировать в приложении статус оплаты: не оплачено, оплачено наличными, переводом, онлайн?
4. Если онлайн-оплата появится позже, какой сценарий ожидается: предоплата полностью, депозит, оплата экипировки отдельно?
5. Нужно ли блокировать бронь без оплаты в будущем?

### 2.11. Постоянные клиенты и лояльность

Лояльность, судя по текущим приоритетам, не входит в MVP, но вопросы стоит сохранить для будущего этапа:

1. Что значит «постоянный клиент»: количество заездов, сумма оплат, ручная отметка администратора?
2. Что именно нужно давать постоянным клиентам: скидку, ранний доступ к слотам, бонусный заезд, бейдж?
3. Лояльность нужна в MVP или это будущий этап?
4. Должен ли клиент видеть свой статус лояльности в приложении?
5. Данные о постоянных клиентах уже есть в бэкенде или их нужно начать собирать с нуля?

### 2.12. API и интеграция

1. Есть ли уже готовый контракт API или его нужно спроектировать в рамках задачи?
2. Какие методы API точно нужны для MVP: авторизация по телефону, список слотов, детали слота, создание брони, отмена брони, список моих броней, получение статуса брони?
3. Какие ошибки должен возвращать бэкенд: мест нет, слот отменён, поздняя отмена запрещена, пользователь не авторизован, бронь отклонена, бронь уже существует?
4. Нужна ли офлайн-поддержка или приложение может работать только при наличии интернета?
5. Как приложение получает изменения по броням и слотам: только при обновлении экрана, через push или через polling?
6. Какие требования к безопасности API: access token, refresh token, HTTPS, защита персональных данных?

## 3. Вопросы, снятые с MVP

Эти вопросы не нужно задавать в рамках текущей MVP-сессии, так как соответствующие функции отложены:

1. фильтры по дате / времени / трассе;
2. выбор своей или прокатной экипировки;
3. оценка маршала после заезда;
4. расширенная программа лояльности;
5. расширенные сценарии оплаты, если оплата не войдёт в MVP.
