# Нефункциональные требования

> Источник: `domain-description.md` — описание домена клиентского мобильного приложения картинг-центра «Апекс». Требования сформированы только из данных домена.

Приоритеты:

- **Must** — входит в MVP / обязательное бизнес-правило.
- **Won’t MVP** — явно отложено или вне скоупа MVP.

| ID | Формулировка | Источник | Приоритет |
|---|---|---|---|
| NFR-001 | Приложение должно использовать существующий бэкенд как источник истины по расписанию, слотам, статусам, доступности и бизнес-ограничениям. | domain-description.md, §1, §15, §16 | Must |
| NFR-002 | Приложение должно работать поверх существующего бэкенда через API-контракт. | domain-description.md, §15, §16 | Must |
| NFR-003 | Клиентское приложение должно корректно отображать данные, полученные от API. | domain-description.md, §1, §15 | Must |
| NFR-004 | Клиентское приложение должно корректно обрабатывать ответы API, включая отказы и недоступность действий. | domain-description.md, §1, §15 | Must |
| NFR-005 | Приложение не должно реализовывать серверную транзакционность, маппинг API на внутренние модели, формирование расписания, назначение маршалов, управление парком картов, отмену слотов администратором и миграцию исторических данных. | domain-description.md, §15 | Must |
| NFR-006 | SLA бэкенда находится вне скоупа клиентского приложения. | domain-description.md, §15 | Must |
| NFR-007 | Административные действия выполняются в существующей инфраструктуре, а не в клиентском мобильном приложении. | domain-description.md, §1, §3, §7, §15 | Must |
