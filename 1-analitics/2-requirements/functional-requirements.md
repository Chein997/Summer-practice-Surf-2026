# Функциональные требования

> Источник: `domain-description.md` — описание домена клиентского мобильного приложения картинг-центра «Апекс». Требования сформированы только из данных домена.

Приоритеты:

- **Must** — входит в MVP / обязательное бизнес-правило.
- **Won’t MVP** — явно отложено или вне скоупа MVP.

| ID | Формулировка | Источник | Приоритет |
|---|---|---|---|
| FR-001 | Приложение должно получать и показывать клиенту слоты на ближайшие 7 дней от текущего момента. | domain-description.md, §6 | Must |
| FR-002 | Если расписания на ближайшие дни нет, приложение должно показывать понятное пустое состояние, например «Пока нет доступных заездов». | domain-description.md, §6 | Must |
| FR-003 | В списке слотов приложение должно показывать дату и время старта, длительность, конфигурацию трассы, уровень, свободные места, цену, статус и адрес картинг-центра. | domain-description.md, §6.1 | Must |
| FR-004 | Слоты без свободных мест должны отображаться со статусом «Мест нет» и без возможности бронирования. | domain-description.md, §6.1 | Must |
| FR-005 | Слоты, отменённые центром, должны отображаться со статусом «Отменён» и без возможности бронирования. | domain-description.md, §6.1 | Must |
| FR-006 | На экране деталей слота приложение должно показывать дату и время старта, длительность, конфигурацию трассы, уровень сложности, свободные места, цену, маршала, адрес центра, место сбора, правила безопасности, условия отмены и кнопку «Забронировать». | domain-description.md, §6.2 | Must |
| FR-007 | В MVP приложение не должно показывать время, к которому нужно приехать заранее. | domain-description.md, §6.2 | Won’t MVP |
| FR-008 | Приложение должно поддерживать вход клиента по номеру телефона с подтверждением через SMS-код. | domain-description.md, §5, §13 | Must |
| FR-009 | Для бронирования приложение должно требовать имя, телефон, email, возраст и согласие с правилами безопасности. | domain-description.md, §5 | Must |
| FR-010 | Если клиенту меньше 18 лет, приложение должно требовать согласие родителя или законного представителя. | domain-description.md, §5 | Must |
| FR-011 | После нажатия «Забронировать» приложение должно отправлять запрос на бронирование в API. | domain-description.md, §7, §13 | Must |
| FR-012 | Приложение должно обрабатывать ответ API: создание брони или отказ, если бронь невозможна. | domain-description.md, §7, §15 | Must |
| FR-013 | После создания брони приложение должно отображать статус «Ожидает подтверждения». | domain-description.md, §7, §8, §13 | Must |
| FR-014 | При подтверждении брони администратором приложение должно отображать статус «Активна». | domain-description.md, §7, §8, §13 | Must |
| FR-015 | При отклонении брони администратором приложение должно отображать статус «Отклонена центром». | domain-description.md, §7, §8, §13 | Must |
| FR-016 | Если администратор долго не подтверждает бронь, приложение должно сохранять отображение статуса «Ожидает подтверждения». | domain-description.md, §7, §14 | Must |
| FR-017 | В приложении должен быть раздел «Мои брони» для просмотра клиентом собственных броней и их актуальных статусов. | domain-description.md, §9 | Must |
| FR-018 | Приложение должно поддерживать статусы брони: «Ожидает подтверждения», «Активна», «Отменена клиентом», «Отменена центром», «Отклонена центром», «Завершена», «Неявка». | domain-description.md, §8 | Must |
| FR-019 | Клиент должен иметь возможность отменить активную или ожидающую бронь, если до старта осталось больше 1 часа. | domain-description.md, §10, §13 | Must |
| FR-020 | Приложение должно запрещать отмену брони, если до старта осталось 1 час или меньше. | domain-description.md, §10 | Must |
| FR-021 | Условия отмены должны показываться клиенту перед бронированием. | domain-description.md, §10 | Must |
| FR-022 | При отмене заезда центром приложение должно показывать статус «Отменена центром», причину отмены, дату и время отменённого заезда и кнопку «Выбрать другой заезд». | domain-description.md, §11 | Must |
| FR-023 | Приложение должно отображать причины отмены центром: погодные условия, техническая неисправность, недоступность трассы, недоступность маршала, организационная причина, другое со свободным текстом. | domain-description.md, §11 | Must |
| FR-024 | Приложение должно поддерживать push-уведомление о подтверждении брони администратором. | domain-description.md, §12 | Must |
| FR-025 | Приложение должно поддерживать push-уведомление об отклонении брони администратором. | domain-description.md, §12 | Must |
| FR-026 | Приложение должно поддерживать push-уведомления-напоминания о заезде за 24 часа и за 2 часа до старта. | domain-description.md, §12 | Must |
| FR-027 | Приложение должно поддерживать push-уведомление об отмене заезда центром. | domain-description.md, §12 | Must |
| FR-028 | В MVP приложение не должно отправлять push-уведомления о создании брони в ожидании подтверждения, отмене брони клиентом и запрете отмены из-за порога в 1 час. | domain-description.md, §12 | Won’t MVP |
| FR-029 | Приложение должно корректно обрабатывать отказ API при отсутствии мест, отменённом слоте, отклонении брони или недоступности действия. | domain-description.md, §15 | Must |
